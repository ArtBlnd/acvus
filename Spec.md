# Acvus Template Engine Specification

## Overview

Acvus는 **패턴 매칭과 이터레이터**라는 단 두 가지 개념 위에 세워진 템플릿 엔진이다.
전통적인 제어문(if, for, while)이 존재하지 않는다. 모든 로직은 패턴 매칭 expression으로 표현되며,
조건 분기와 반복이 동일한 메커니즘으로 통합된다.

## Design Principles

1. **Statement가 없다** — 모든 `{{ }}`는 expression이며, Rust 블럭처럼 자체 output을 가진다.
2. **패턴 매칭이 곧 제어 흐름** — if/for/while 없이 패턴 매칭만으로 분기와 반복을 표현한다.
3. **모든 것은 이터레이터** — 함수의 반환값은 이터레이터다. 단일 값 매칭은 0-or-1 원소 시퀀스로 통합된다.
4. **함수형 합성** — 파이프 연산자 `|`와 고차 함수로 데이터를 변환한다.
5. **최소 문법** — syntax sugar로 일상적 패턴을 간결하게 유지한다.
6. **모든 변수는 immutable** — `=`는 항상 새 바인딩이다. mutable 변수는 존재하지 않는다.

## Syntax

### Literal Text

`{{ }}` 바깥의 텍스트는 그대로 출력된다.

```
안녕하세요, 반갑습니다.
```

→ `안녕하세요, 반갑습니다.`

### Expression

모든 `{{ }}`는 expression이다. 평가 결과가 곧 output이 된다.

```
{{ "hello" }}          → hello
{{ 1 + 2 }}            → 3
{{ name }}             → (name의 값)
```

별도의 output이 없는 expression은 빈 문자열 `""`을 반환한다.

### Pipe Operator

`|`로 함수를 체이닝한다. 왼쪽 값이 오른쪽 함수의 첫 번째 인자로 전달된다.

```
{{ list | filter(x -> x != 0) | map(x => x * 2) }}
```

### Lambda

`->` 화살표 문법으로 익명 함수를 정의한다.

```
x -> x != 0
(x, y) -> x + y
```

## Pattern Matching

Acvus의 핵심. `=` 연산자로 왼쪽 패턴을 오른쪽 expression에 매칭한다.

```
{{ pattern = expression }}
```

하나의 규칙: **패턴 내 상수는 필터하고, 변수는 캡처한다.**

`=`는 항상 새 바인딩을 생성한다. 모든 캡처된 변수는 immutable이다.

### 실행 모델

모든 `{{ pattern = expr }}` 블럭은 동일한 메커니즘으로 동작한다:

1. `expr`을 평가하여 이터레이터를 얻는다.
2. 이터레이터의 각 원소에 대해 `pattern`을 매칭한다.
3. 매칭 성공 시, 패턴 내 변수들이 캡처되고 바디가 렌더링된다.
4. 매칭 실패 시 (또는 이터레이터가 비어있으면), `{{_}}`로 넘어간다.
5. `{{/}}`로 블럭을 닫는다.

**조건 분기와 반복은 같은 메커니즘이다.**
단일 값 매칭은 0-or-1 원소 이터레이터, 리스트 바인딩은 N 원소 이터레이터일 뿐이다.

### 패턴의 종류

**변수 패턴** — 값을 캡처한다. 항상 매칭 성공.

```
{{ item = list }}
  {{item}}
{{/}}
```

각 원소가 `item`에 캡처되어 바디가 원소 수만큼 반복 렌더링된다.

**상수 패턴** — 값을 필터한다. 캡처하지 않는다.

```
{{ true = is_valid }}
  유효합니다.
{{/}}
```

`is_valid`가 `true`이면 매칭 성공, 바디 1회 렌더링. 캡처할 변수가 없으므로 바디 내에서 참조할 것은 없다.

**복합 패턴** — 상수로 필터하고, 변수로 캡처한다.

```
{{ [true, a] = pair }}
  {{a}}
{{/}}
```

첫 번째 원소가 `true`인지 필터하고, 두 번째 원소를 `a`에 캡처한다.

**Destructuring 패턴** — 리스트를 분해하여 각 원소를 캡처한다.

```
{{ [a, b, c, ..] = list }}
  첫째: {{a}}, 둘째: {{b}}, 셋째: {{c}}
{{/}}
```

`..`은 나머지 원소를 무시한다. 리스트 길이가 패턴보다 짧으면 매칭 실패.

### Multi-arm

하나의 expression에 대해 여러 패턴을 순차적으로 매칭할 수 있다.
첫 arm에서 `{{ pattern = expr }}`로 매칭 대상을 지정하고,
이후 arm에서는 `{{ pattern = _ }}`로 동일한 expression에 대한 추가 arm을 작성한다.

```
{{ "admin" = role }}
  관리자 페이지
{{ "user" = _ }}
  사용자 대시보드
{{_}}
  게스트 뷰
{{/}}
```

- `{{ pattern = _ }}` — expression 위치의 `_`는 "이전 arm의 매칭 대상"을 참조
- `{{_}}` — catch-all arm. `=` 없이 패턴 위치에 `_`만 사용

이 둘은 문법적으로 완전히 구분된다.

### Catch-all `{{_}}`

`{{_}}`는 이터레이터가 비었거나 이전 패턴이 모두 실패했을 때 실행된다.

```
{{ true = is_valid }}
  유효합니다.
{{_}}
  유효하지 않습니다.
{{/}}

{{ item = list }}
  {{item}}
{{_}}
  리스트가 비어있습니다.
{{/}}
```

두 경우 모두 `{{_}}`의 의미는 동일하다: **이터레이터가 비었을 때.**

## Null과 타입 안전성

Acvus는 **런타임 에러가 없다.** 모든 에러는 컴파일 타임에 잡힌다.

- **null** (값이 없음, 타입은 있음) → `[]` 빈 이터레이터로 표현. 패턴 매칭으로 자연스럽게 처리.
- **타입 없음** (필드/타입 자체가 존재하지 않음) → 컴파일 에러.

```
{{-- b가 nullable일 때: 런타임에 null이면 [] → {{_}}로 --}}
{{ x = a.b.c }}
  {{x}}
{{_}}
  값이 없습니다.
{{/}}

{{-- a의 타입에 z 필드가 없으면: 컴파일 에러 --}}
{{ x = a.z }}
```

null은 조건, 반복과 동일한 메커니즘으로 처리된다:

| 상황 | 이터레이터 | 결과 |
|---|---|---|
| null 값 | `[]` | `{{_}}`로 |
| non-null 단일 값 | `[value]` | 블럭 1회 실행 |
| 리스트 | `[a, b, c, ...]` | N회 반복 |
| 패턴 매칭 실패 | `[]` | `{{_}}`로 |

**null, 조건 분기, 반복, 패턴 매칭 실패가 전부 동일한 메커니즘이다.**

## Storage Reference (`$`)

`$`는 외부 스토리지에 대한 레퍼런스를 나타낸다. 모든 변수는 immutable이지만,
`$`는 스토리지 slot에 값을 쓸 수 있다.

### 규칙

1. **`$`는 `$`에서만 올 수 있다** — 무에서 `$` 변수를 생성할 수 없다. 반드시 외부에서 제공된 `$`로부터 파생되어야 한다.
2. **`{{ $name = expr }}`는 스토리지 쓰기** — `$name`이 가리키는 스토리지 slot에 `expr`의 값을 쓴다.
3. **Destructuring으로 `$` 전파** — `$` 변수를 destructure할 때, `$`로 꺼내면 레퍼런스, `$` 없이 꺼내면 값 복사.

### 읽기

```
{{ $global }}                ← 스토리지 값 출력
{{ $global.field }}          ← 필드 값 출력
```

### 쓰기

```
{{ $global = expr }}         ← 스토리지 전체에 쓰기
```

### Destructuring을 통한 부분 쓰기

```
{{ { $value, name } = $global }}
  {{ $value = 42 }}          ← $global.value에 쓰기 ($에서 파생된 $)
  {{ name }}                 ← immutable 복사본, 읽기만
{{/}}
```

`$`로 destructure한 필드는 원본 스토리지의 해당 필드에 대한 레퍼런스가 된다.
`$` 없이 destructure한 필드는 값의 복사본이다.

## Inline Expression

바인딩 없이 expression만 쓰면 결과가 바로 출력된다.

```
{{ name }}
{{ list | filter(x -> x != 0) }}
{{ "hello" }}
```

`{{ }}` 안에 `=`가 없으면 inline expression이다.

---

## Nesting

블럭은 자유롭게 중첩된다.

```
{{ user = users }}
  이름: {{ user.name }}
  {{ post = user.posts }}
    - {{ post.title }}
  {{/}}
{{/}}
```

## Syntax Summary

| Syntax | Description |
|---|---|
| `text` | 리터럴 텍스트, 그대로 출력 |
| `{{ expr }}` | expression 평가 후 출력 |
| `{{ pattern = expr }}...{{/}}` | 패턴 매칭 + 이터레이션 블럭 |
| `{{ pattern = _ }}` | multi-arm: 이전 매칭 대상에 대한 추가 arm |
| `{{_}}` | catch-all arm (이터레이터가 비었을 때) |
| `{{/}}` | 블럭 닫기 |
| `{{ $name = expr }}` | 스토리지 쓰기 |
| `{{ { $a, b } = $x }}` | destructuring: $a는 레퍼런스, b는 값 복사 |
| `expr \| func(args)` | 파이프 연산자 |
| `x -> expr` | 람다 expression |
| `[a, b, ..]` | destructuring 패턴 |
| `{{-- comment --}}` | 주석 (출력 없음) |

## Design Decisions

- **모든 변수는 immutable**: `=`는 항상 새 바인딩. mutable 변수 없음. `:=` 불필요.
- **`$`는 스토리지 레퍼런스**: `$`는 `$`에서만 파생. 무에서 생성 불가. destructuring으로 전파.
- **Multi-arm 패턴 매칭**: `{{ pattern = _ }}`로 이전 매칭 대상을 참조. `{{_}}`(catch-all)와 문법적으로 구분됨.
- **인덱스 접근**: 별도 문법 없음. `enumerate` 같은 함수로 처리.
- **include/import**: 존재하지 않음. 단일 템플릿이 완결적이다.
- **whitespace 처리**: 별도 정책 없음. `indent(n)` 등 함수로 제어.
- **enum**: 템플릿 레벨에서 지원하지 않음. 호스트가 데이터를 평탄화하여 전달. 필요 시 패턴 문법 확장으로 대응 가능.
- **함수**: 함수 정의는 호스트 사이드에서만 등록. 템플릿에서는 파이프 `|`와 람다 `->`를 통한 컴포지션만 가능.

## Range Pattern

범위 매칭을 위한 세 가지 range 문법을 지원한다.

| 문법 | 의미 | 범위 |
|------|------|------|
| `0..10` | exclusive end | [0, 10) |
| `0..=10` | inclusive end | [0, 10] |
| `0=..10` | exclusive start | (0, 10] |

expression과 pattern 양쪽에서 사용 가능하다.

```
{{-- Range pattern --}}
{{ 0..10 = age }}
  어린이
{{ 10..=19 = _ }}
  청소년
{{_}}
  성인
{{/}}

{{-- Range expression --}}
{{ 1..=5 }}
```

## Compilation Pipeline

Acvus는 인터프리터 없이 컴파일 방식으로 동작한다. 백엔드 타겟은 미정.

```
Template → AST → Type Check → MIR → Backend (미정)
```

- **AST**: 구문 레벨 표현 (파서 출력)
- **Type Check**: 타입 추론 및 검증
- **MIR**: 타입이 붙은 linear IR. 패턴 매칭 lowering, 파이프 desugar, 클로저 캡처 분석 포함.
- **Optimization**: MIR → MIR 변환 패스.
- **Backend**: MIR에서 최종 타겟으로 코드 생성. 후보: WASM, 네이티브, JS 등.

### 컴파일 철학: Correct First, Fast Later

MIR 생성은 **정확성 우선**으로 한다. 초기 변환은 naive하게 의미만 보존하고, 최적화는 별도 패스에서 수행한다.

#### 호출 구분

MIR은 두 종류의 호출을 구분한다:

- **`call`**: 빌트인 함수. 동기 실행.
- **`async_call`**: 외부 함수. 비동기 실행. `await` 필요.

초기 MIR 생성 시 외부 호출은 전부 `async_call`로 변환된다. 최적화 없이 직관적으로 변환한다.

#### 최적화 패스 (MIR → MIR)

정확한 MIR이 생성된 후, 별도의 최적화 패스가 변환을 수행한다:

- 독립적인 `async_call`을 배치로 묶기
- pure 함수의 `map` → `pmap` 자동 승격
- 인라이닝 가능한 `call` 인라인
- 기타 최적화

## Type System

### 원칙

- **모든 타입은 컴파일 타임에 결정된다.** 동적 타입은 존재하지 않는다.
- 변수의 타입은 항상 RHS expression의 타입으로 결정된다.
- Storage(`$`) 변수의 타입은 외부에서 제공된다.
- 유저가 타입을 명시할 필요가 없다. 전부 추론된다.

### 타입 종류

| 타입 | 예시 |
|------|------|
| `Int` | `42` |
| `Float` | `3.14` |
| `String` | `"hello"` |
| `Bool` | `true`, `false` |
| `List<T>` | `[1, 2, 3]` |
| `Object<{...}>` | `{ name, value }` |
| `Fn(A, B) -> C` | `x -> x + 1` |

### 설계 결정

| 결정 | 값 | 이유 |
|------|-----|------|
| 서브타이핑 | 없음 | 복잡도 회피 |
| 유저 정의 제네릭 | 없음 | 파이프 + 빌트인이 암묵적으로 제공 |
| HKT | 없음 | 불필요 |
| Mutation | 없음 | 모든 바인딩은 immutable |
| 라이프타임 | 없음 | 소유권 모델 불필요 |
| 리스트 | Homogeneous | `[1, "a"]`는 컴파일 에러. Heterogeneous 허용 시 런타임 태그드 유니온 필요. |
| 추론 방식 | 단방향 + 빌트인 unify | 빌트인 함수만 내부적으로 타입 변수 사용 |

### 타입 규칙

#### 산술 연산
- **`Int op Int` → `Int`**, **`Float op Float` → `Float`**. 혼합 연산(`Int + Float`)은 **타입 에러**.
- **`Int / Int` → `Int`** (truncate). 정수 나눗셈은 항상 버림.
- 명시적 변환이 필요: `n | to_float`, `x | to_int`.

#### Null과 List 통합
- 별도의 `Option<T>` 타입은 존재하지 않는다.
- Null은 `List<T>`의 빈 리스트 `[]`로 표현된다.
- 단일 값은 `[value]` (1-원소 리스트)로 취급된다.

#### 빈 리스트
- `[]`의 타입은 사용 컨텍스트에서 추론된다.
- 컨텍스트 없이 `[]` 단독 사용은 타입 에러.

#### Object 매칭
- Object는 **open (서브셋 매칭)**이다.
- `{ name }` 패턴은 `{ name, age, email }` 오브젝트와 매칭된다.
- Rust의 `..`처럼 나머지 필드를 명시할 필요 없다.

#### 동등성 비교 (`==`)
- **모든 타입에서 구조적 비교(deep equality)** 지원.
- `List == List`: 원소별 비교. `T`가 `==` 가능하면 `List<T>`도 가능.
- `Object == Object`: 필드별 비교.
- Mutation 없음 + 순환 참조 불가능이므로 구조적 비교는 항상 sound.

#### Range 바운드
- **`Int`만 허용**. `Float` range는 타입 에러.

#### Emit (출력)
- `{{ expr }}`로 출력할 수 있는 타입은 **`String`만** 가능.
- 다른 타입은 명시적 변환 필요: `{{ count | to_string }}`.
- 타입 안전성을 위해 암묵적 `to_string` 변환은 존재하지 않는다.

#### Exhaustiveness
- **`{{_}}` catch-all은 필수**. 모든 매치 블럭에 `{{_}}`가 있어야 한다.
- 정적 exhaustiveness 분석이 불필요해진다. 컴파일러는 `{{_}}` 존재 여부만 검사.

#### Storage 타입
- Storage(`$`) 변수의 타입은 호스트가 외부에서 제공한다.
- 선언 형식은 미정.

### 빌트인 제네릭

유저 정의 제네릭은 없지만, 빌트인 함수는 내부적으로 제네릭 시그니처를 가진다.
타입 체커가 호출 시점에 unification으로 구체 타입을 결정한다.

```
filter: (List<T>, Fn(T) -> Bool) -> List<T>
map:    (List<T>, Fn(T) -> U) -> List<U>
```

유저는 타입 변수를 볼 일이 없다. 파이프가 암묵적으로 제네릭을 제공한다.

## Async & Parallelism

### 원칙

- **외부 함수 호출은 모두 async로 취급**한다.
- 빌트인 함수(산술, `filter`, `map` 등)는 내부에서 동기 실행된다.
- Mutation이 없으므로 순수 함수는 컴파일러가 자동으로 병렬화할 수 있다.

### 함수 등록

호스트가 함수를 등록할 때 `pure/effectful` 여부를 마킹한다.

```
register("fetch_user",  effectful, async, ...)
register("to_upper",    pure,      sync,  ...)
```

### `map` vs `pmap`

| 함수 종류 | `map` | `pmap` |
|-----------|-------|--------|
| **pure** | 옵티마이저가 자동 병렬화 | 동일 결과 |
| **effectful** | 순차 실행 (안전한 기본값) | 유저가 명시적으로 병렬화 (유저 책임) |

- `map`: 기본. pure 함수는 옵티마이저가 알아서 병렬화. effectful 함수는 순차.
- `pmap`: 유저가 "이 effectful 함수는 병렬해도 안전하다"고 명시적으로 선언.
- `pmap`은 eager collect 후 병렬 실행. 본질적으로 eager이다.

```
{{-- 순차 실행 --}}
{{ user_ids | map(id -> fetch_user(id)) }}

{{-- 유저가 명시적으로 병렬화 --}}
{{ user_ids | pmap(id -> fetch_user(id)) }}
```

### 실행 모델

실행은 코루틴 방식으로 동작한다:

```
실행
  → 독립적인 외부 호출을 배치로 모음
  → yield (호스트에 위임)
  → 호스트가 병렬 실행
  → 결과 전달
  → 재개
```
